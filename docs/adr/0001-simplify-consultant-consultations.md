# Simplify consultant around consultations

The current consultant skill is too close to Wayfinder: it exposes tickets, moves, gates, and map terminology even though the desired experience is closer to `/grill-me` with persistence. Rebuild `/consultant` as a single `SKILL.md`: delete the current reference files and inline the tiny templates/prompts. The runtime process should center on one live `consultation.md`, optional `references/`, and a final `plan.md`; kill separate ticket files and hide move terminology from the client-facing workflow; do not create `implementation-notes.md` or include during/post-implementation routines. Startup stays slug-based but low-ceremony: explicit workspace paths resume, similar existing slugs ask continue/new, and otherwise a new `~/.consultant/<slug>` is created. The skill should be general to downstream agent work, not coding-only: the builder may be coding, researching, writing, operating systems, or doing any other concrete execution. The consultant should inspect discoverable facts in the territory before asking the client; questions are for taste, priorities, constraints, and choices the client owns. The live file should stay tiny while explicitly naming `Brief`, `Known knowns`, `Unknown knowns`, `Known unknowns`, `Unknown unknowns`, `Decisions`, and `Next step`, and it should be updated after every meaningful decision or unknowns change before the consultant asks the next question. The final plan should be a lean builder brief: outcome, key decisions, implementation shape, constraints, references, out of scope, and risks/open assumptions. Material unknowns that could change the builder's output must be resolved before handoff; minor or explicitly client-accepted assumptions may remain as risks/open assumptions. References/prototypes should be created selectively, only when concrete options reveal taste or unknowns better than another verbal question. Every important client-facing question should default to 2–4 clear options, include the consultant's recommendation, and leave room for the client to choose something else. Before handoff, run a mandatory adversarial subagent review of the draft plan and explicitly referenced files, excluding `consultation.md`; review the findings with the client, iterate, and repeat until the reviewer approves or the client explicitly chooses to exit anyway. This preserves durable handoff value while making the loop feel like a consultant asking the next most important question, offering clear options, recommending one, recording the decision, and moving on.

## Considered Options

- Separate ticket files for every decision — rejected because they make the skill feel like a project-management system.
- Explicit named moves such as blind-spot pass, options, interview, research, and task — rejected for the client-facing flow because they leak process vocabulary.
- A heavyweight unknowns taxonomy in `consultation.md` — rejected; the file should explicitly name the four quadrants, but the consultant fills them on the client's behalf and keeps them terse.
- Creating prototypes/references as a standard phase — rejected because it would make the skill heavy; they are reserved for cases where the client will know the answer better by reacting to something concrete.
- A PRD-style or file-by-file implementation plan as the default `plan.md` — rejected because it either over-formalizes the artifact or over-specifies implementation before the builder meets the territory.
- Optional or informal plan review before handoff — rejected because the final artifact should survive adversarial scrutiny by a fresh agent before the consultation exits.
- A single consultation save file — preferred because it keeps the process simple enough to run every time while still giving future agents a durable artifact.
